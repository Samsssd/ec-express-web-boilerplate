# Excelsior app — agent contract

You are editing a live, user-facing web app generated by the Excelsior platform.
The stack: **Next.js 15 (App Router) + TypeScript + Tailwind CSS v4**. The dev
server is already running with HMR — your edits appear in the user's preview
instantly. Verification happens in the preview: **never** run `npm run build`,
`tsc`, `eslint` or tests (they are blocked anyway).

**All user-facing text is French.** Code, comments and identifiers stay English.

## Architecture rules

- **Reads happen in Server Components**: `const supabase = await createClient()`
  (from `@/lib/supabase/server`) → `await supabase.auth.getUser()` → query.
- **Writes are Server Actions** (`"use server"`): re-check `getUser()`, mutate,
  then `revalidatePath(...)`. No API routes for app CRUD, no client-side data
  stores for server data (zustand is for pure UI state only).
- Client components are leaf islands for interactivity; never fetch Supabase
  data client-side when a Server Component can pass it down.

## Auth (pre-wired — Supabase, cookie SSR)

- Sessions live in cookies via `@supabase/ssr` + `middleware.ts`. **No client
  auth store, no AuthProvider, no `onAuthStateChange`.**
- Email confirmation is DISABLED on the shared project: `signUp()` returns an
  active session immediately. **Never** build a "vérifiez votre e-mail" /
  magic-link / OTP flow, never set `emailRedirectTo`.
- Server-side user read: use `getCurrentUser()` from `@/lib/supabase/auth`
  (internally backed by `getUser()`, never `getSession()`). When a page/action
  already created a Supabase client for queries, pass it as
  `getCurrentUser(supabase)`. A protected page redirects to `/auth` when it
  returns `null`; transient Supabase failures are surfaced instead of being
  misreported as a logout.
- `signIn`/`signUp` are `useActionState` reducers
  (`(prevState, formData) => Promise<AuthState>`); `signOut` is a no-arg action
  wired as `<form action={signOut}><button>Déconnexion</button></form>`.
- Owned rows: stamp `user_id: user.id` on inserts and filter reads by it (RLS
  also enforces it). Read "self" rows with `.maybeSingle()` and treat missing
  as "create it".

## Base de données — schema changes via migrations ONLY

The app talks to a **shared** Supabase project with the public anon key. The
schema is managed through SQL files in `supabase/migrations/`:

- **Workflow**: to change the schema, write a NEW file
  `supabase/migrations/NNNN_slug.sql` (zero-padded, ordered — `0001_users.sql`,
  `0002_orders.sql`, …). The platform applies new files automatically after
  your turn. **Never run SQL yourself** (no psql exists here), and **never edit
  a file that was already applied** — applied files are tracked by content
  hash; always create a new file.
- **Naming**: every table is prefixed with the app id — read it from
  `NEXT_PUBLIC_APP_ID` in `.env.local` and use its literal value, e.g.
  `app_ab12cd_orders`. Unprefixed names are rejected.
- **Idempotent DDL only**: `CREATE TABLE IF NOT EXISTS`,
  `CREATE INDEX IF NOT EXISTS`, `ALTER TABLE`, and `DROP POLICY IF EXISTS`
  before every `CREATE POLICY` (Postgres has no `CREATE POLICY IF NOT EXISTS`).
- Keep each file below **64 KB and 40 SQL statements**.
- **Every new table, in the same file**: `ENABLE ROW LEVEL SECURITY`,
  owner-scoped policies (`auth.uid()`), and
  `GRANT SELECT, INSERT, UPDATE, DELETE ON <table> TO anon, authenticated;`.
- **Every web policy** must compare an owner column to `auth.uid()`; adding one
  safe policy does not make a second permissive policy safe. `USING (true)` is
  rejected for web apps.
- **Forbidden** (rejected by the platform validator): functions, triggers,
  extensions, roles, `DROP TABLE`, `TRUNCATE`, `DO $$` blocks,
  `SECURITY DEFINER`, and touching other apps' tables or `auth.*`/`storage.*`.
  The only `auth.*` exception is an FK `REFERENCES auth.users (id)`.
- The reserved users profile table is `<appId>_users` (`id` = the auth user
  id). `app/actions/auth.ts` already upserts it on sign-in/sign-up — it just
  tolerates the table not existing yet. **Your first migration for any
  DB-backed feature should create it.** Canonical example (`0001_users.sql`,
  replace `app_ab12cd` with the real app id):

```sql
create table if not exists app_ab12cd_users (
  id uuid primary key references auth.users (id) on delete cascade,
  email text,
  full_name text,
  created_at timestamptz not null default now()
);
alter table app_ab12cd_users enable row level security;
drop policy if exists "users_select_own" on app_ab12cd_users;
create policy "users_select_own" on app_ab12cd_users
  for select using (id = auth.uid());
drop policy if exists "users_write_own" on app_ab12cd_users;
create policy "users_write_own" on app_ab12cd_users
  for all using (id = auth.uid()) with check (id = auth.uid());
grant select, insert, update, delete on app_ab12cd_users to anon, authenticated;
```

Owned child tables scope by `user_id = auth.uid()`; a child table without
`user_id` scopes through its FK to an owned parent (EXISTS subquery in the
policy). FKs reference other `<appId>_` tables or `auth.users` — never a bare
unprefixed name.

## Paiements — hosted Stripe Checkout via the platform proxy

- The app holds **no Stripe key**. Never `import Stripe from "stripe"`, never
  `@stripe/stripe-js` (not installed). Payment = hosted checkout redirect only.
- From a **Server Action**: `createCheckoutSession(input)` from
  `@/lib/payments/checkout` → redirect the user to the returned `url`.
  Check the result with `getCheckoutStatus(sessionId)` on the success page.
- Prices are server-side constants (cents). Use `formatPrice()` from
  `@/lib/payments/money` for display. Never trust a price from the client.

## Fichiers — S3 via presigned tickets

- Client island: call the `getUploadTicket(input)` Server Action
  (`@/app/actions/upload`), then `uploadToBucket(ticket, file)` from
  `@/lib/storage/upload-client`, then persist the returned public `url` via
  your normal create/update action.
- Limits are enforced by the platform (10 MB, images/PDF/audio/video). Never
  `import @aws-sdk/*` (not installed), never ask for AWS credentials.

## IA — streaming chat via the platform proxy

- `app/api/chat/route.ts` exists and streams. Client-side: POST
  `{ messages, model }` to `/api/chat` and read the plain-text stream.
- Models are the aliases `"fast" | "balanced" | "thinking"` — never raw model
  ids, never provider SDKs, never an API key.

## Environment

`.env.local` is injected by the platform (`NEXT_PUBLIC_SUPABASE_URL`,
`NEXT_PUBLIC_SUPABASE_ANON_KEY`, `NEXT_PUBLIC_APP_ID`, `PAYMENTS_API_*`,
`STORAGE_API_*`, `AI_API_*`). Never edit it, never ask the user for API keys,
never hardcode secrets. If a capability seems unconfigured, say so instead of
inventing credentials.
